# Автообновление в облаке (Фаза 5)

Цель — репозиторий обновляется сам, без постоянно включённого ПК. Реализовано двумя
GitHub Actions workflow (выполняются в облаке GitHub, бесплатно для приватных репо в
пределах лимитов Actions).

## A. `sync-descriptors.yml` — надёжный синк (рекомендуется включить сразу)

Триггер: push в `master`, меняющий манифесты/`asset_index.json`/`aggregate.py`, или ручной
запуск. Делает: пересборку `descriptors/` + `catalog.json`, публикацию `asset_index.json` на
HF, коммит дескрипторов. **GDrive не нужен**, работает только с содержимым репозитория.

Требуется один секрет:
- `HF_TOKEN` — write-токен Hugging Face (для публикации индекса на `Artyudin/sr-mods-assets`).

Без `HF_TOKEN` шаг публикации индекса просто пропускается (дескрипторы всё равно пересоберутся).

## B. `auto-update.yml` — плановый полный прогон (проверен в облаке, расписание ВКЛЮЧЕНО)

Триггер: ежедневно 03:00 UTC (06:00 МСК) + ручной запуск (параметры `only`/`force`/`notify_test`).
Делает (`--cloud`): инкрементально тянет изменённые юниты с Google Drive → декомпиляция →
манифесты → ассеты и код на HF → дескрипторы → push. Затем шлёт **Telegram-итог** обновлённых
паков. Раздельные шаги: ранний push детекции (доказательство сохраняется, даже если тяжёлый
HF-шаг споткнётся), HF-шаги `continue-on-error`, финальный push с fetch+rebase+ретраем.

Секреты:
- `REPO_PAT` — Personal Access Token (scope `repo`) для checkout приватного `rson-decompiler`
  и push в `sr-mods-aggregator`. Создать: GitHub → Settings → Developer settings → PATs.
- `HF_TOKEN` — write-токен Hugging Face.
- `RCLONE_CONF` — содержимое `rclone.conf` с remote `gdrive` (OAuth refresh-token), БЕЗ BOM.
  Взять из файла `rclone config file`.
- `TG_BOT_TOKEN`, `TG_CHAT_ID` — Telegram-бот и chat_id для уведомлений (опц.; без них
  уведомления пропускаются). chat_id берётся через @userinfobot; боту нужно один раз нажать Start.

### Тяжёлые (manual) юниты
Юнит с `manual:true` (напр. `redux_base_installer` ~12 ГБ — не влезает в раннер) в облаке
**не качается**: пайплайн только детектит изменение по `remote_sig` → `state/manual_pending.json`
→ **Telegram-алерт** «нужен локальный прогон». Обновляешь локально: запусти **`update_base.bat`**
(скачать → декомпиляция → HF → дескрипторы → push, в конце пауза). Нужен env `HF_TOKEN`.

### Уведомления (`pipeline/notify.py`, UTF-8)
- режим `manual` — алерт о тяжёлом юните (ранний шаг);
- режим `summary` — итог обычного автообновления «обновлено паков: …» (финальный шаг);
- `workflow_dispatch` с `notify_test=true` шлёт тестовые сообщения обоих видов.
- Сообщение шлётся **из облака** (важно: `api.telegram.org` может резаться сетью в РФ, но из
  GitHub-раннера доходит; приложение Telegram принимает по MTProto).

### Ограничения (важно)
- **Диск раннера ~14 ГБ.** Юнит крупнее (например `redux_base_installer` ~12 ГБ) может не
  влезть даже с `--lean`. Решение для крупных юнитов — **self-hosted runner** на дешёвом
  всегда-включённом VPS (тогда диск/время не ограничены), либо запуск такого юнита вручную с
  локальной машины.
- **Лимит задания 6 ч.** Прогон без изменений — минуты (skip-if-unchanged по `state/lock.json`
  и `remote_sig`). Крупное изменение может не уложиться.
- **Push воркфлоу-файлов** требует у токена scope `workflow`.
- `git push` из workflow A не триггерит workflow B (стандартная защита Actions от рекурсии) —
  это и не нужно, B идёт по расписанию.

### Статус (боевой)
Оба workflow проверены в облаке end-to-end, секреты заданы (Actions Secrets), **расписание B
включено** (`schedule: cron '0 3 * * *'`). Подтверждено на реальном обновлении: облако само
обнаружило изменённые `redux_fixes` и `universe_fixes_130325`, обработало и выложило; Telegram-
уведомления (manual-алерт и summary-итог) доходят с читаемым текстом.

- **A (`sync-descriptors`)** — пропускает коммиты автора `sr-mods-bot` (чтобы не гоняться с B).
- **B (`auto-update`)** — `--cloud`; тяжёлый `redux_base` идёт через Telegram + `update_base.bat`.
- **Выключить расписание** — закомментировать блок `schedule:` в `auto-update.yml`.

### Если обновится тяжёлый юнит на бесплатном раннере
Диск ~14 ГБ: только `redux_base` (~12 ГБ) не влезает — он `manual:true` (локальный прогон по
Telegram-уведомлению). Если захочешь 100% автоматизации и для него — подключи **self-hosted
runner** на дешёвом всегда-включённом VPS/мини-ПК (диск/время не ограничены).
